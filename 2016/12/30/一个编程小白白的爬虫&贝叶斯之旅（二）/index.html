一个编程小白白的爬虫&贝叶斯之旅（二）

12月 30, 2016 发布在 python

说完了耗时最长的爬虫，接下来我们说说后面的部分，这个时候的小白白已经不是最开始那个纯纯小白了，对python的熟悉了很多，再加上后面的工作主要是逻辑性大于耗时性，所以之后的预处理、训练和分类用的时间和爬虫总共用的时间差不多，我们就按照时间分布把后面三部分放在一起说啦~

小白白童鞋在爬了将近20万的文章，生成了将近20万的文本文件之后，被同学警告说这样的频繁文件读取还都是小文件会对硬盘有损伤的，本着将信将疑的态度百度了一下，确实大部分的说法都说对磁头有一定的损伤，所以最后本着心疼硬盘的态度（其实还是想了一下万一，然后就心疼换硬盘的钱）还是决定转战数据库了，毕竟之后还要对这将近20万篇文章进行分词，去停用词，存储，建立数据词典，存储，想想文件读取就为硬盘难过。数据库经过比较选择了redis，后来真的为这个决定感到无比的正确，因为redis真的很轻量很好用，强烈安利一下～ 下面进入正题~

第二步：对语料库进行预处理

（1）搭建redis数据库

网上的redis教程以Linux系统的偏多，这里就要感叹一下，Linux系统开发必备，有空小白白要去装个双系统，研究下Linux系统，在这里立个flag！ 菜鸟教程上有windows系统上安装redis数据库的详细过程。安装过程其实非常简单，解压文件夹后，在该路径下用redis-server.exe redis.windows.conf命令启动服务器，再用redis-cli.exe -h 127.0.0.1 -p 6379启动另一个cmd作为客户端，过程这里就不再赘述了。

注：打开cmd最好使用管理员权限运行，否则可能会报错，因为某些安装目录普通用户不具有相应权限

因为要向redis数据库中导入十万级的数据，而且小白真的不适应命令行界面，尤其是想查看所有的词和它们的词频时，于是小白就坚定了用redis图形化工具的想法。在网上查了一下，貌似redis的图形化工具并不多，总结起来比较好的有三个，这里是参考网址，redis desktop manager, redis client, redis studio, 综合使用性能和自己的需求之后我选了redis desktop manager，下载安装什么的就不说了，连接数据库的时候填上name,host,port就好了，name自己取，host就是127.0.0.1，,端口号一般也不会改，默认为6379。注意代表服务器的cmd要一直开着，否则是连不上的。（个人认为图形化工具应该相当于一个客户端，隐藏了向服务器发送命令和获取原始形态数据的过程，将原始数据进行整理并显示到图形界面上）

python中连接redis的代码也只用一行就足够r = redis.Redis(host='localhost',port=6379,db=0)

第三步：构建数据词典

对爬取的文章进行预处理了之后，它们就不再是完整的文章、完整的句子了，而是一个个不语义上那么衔接的词，就像上图一样，这也是接下来我们要处理的形式。接下来我们要做的工作是统计词频，简单地说就是分类别统计该类别的所有文章中每个词出现了多少次，最后给出不同类别中各个词出现的频度，这也就是传说中的训练过程。

这里有一个问题要重点说下，就是训练结果的数据类型选择，与之前的list不同，训练结果的数据构成为“词+词频”，考虑到之后要以词本身作为索引对象查找数据库表，因此选择了hash表的结构存储训练的结果。

这里要着重说明的是getkeywords()函数。因为20万篇文章的数据集真的是不小（至少对于我这样的小白白来说），如果按照原理直接分词必然会导致数据词典的维数过高，而且我们的目的是为了找出与类别关联度较大的词，也就是出现频数较高的词，越是这样的词就越有更大的可能性与当前类别相关。不同的文章讲的内容不一样，用词也就会有很大的不同，对于那些在某篇文章中仅出现几次的词与其文章主题的关联很大可能上并不大，这也是我们去停用词的初衷，因其与文章主题无甚关联。所以我们规定，训练时只取每篇文章频数前15的关键词，这也是getkeywords()做的事情。

第四步：对测试集分类

终于进行到了最后一步，这真的是激动人心的一步，因为之前所做的所有处理都会在这一步检测他们的效果，最后能不能成功分类，会不会有预想的效果，准确率会是多少，都会在这一步揭晓，有木有很忐忑又很期待~

我知道我下面应该快快说正题~

分类的原理就是对测试集每篇文章（预处理过的）的所有词在某一类别中的词频都找到，然后输入贝叶斯公式，计算出当前文章属于当前类别的概率，比较当前文章属于N个类别的概率，取最大概率属于的类别为最终类别。

关于贝叶斯公式的选择：朴素贝叶斯的文本分类有两种模型，多项式模型和伯努利模型，两种模型的原理和例子我参考了一篇博客，根据数据统计的难易程度，我选择了多项式模型，于是就有了公式如下：

其中先验概率P(Vj)=当前类下单词总数/整个训练样本的单词总数

p.s.如果还用了其他降维方式，这个类下单词总数就并不那么好算了，这也是小白白没有再继续降维的原因。单词总数都是按照允许重复的原则计算的，因此按照每篇文章15个词，再乘以文章数就好了，其实真正算的时候相当于该类文章数占总文章数的比率。

类条件概率P(ai|Vj)=(当前类别的数据词典中某单词ai的频数+1)/(当前类下单词总数+|V|)

V是训练样本的单词表（即抽取单词，单词出现多次，只算一个），|V|则表示训练样本包含多少种单词，分子中的+1和分母中的+|V|都是针对零概率的平滑处理，不然连乘中如出现某一乘数为0，将是对连乘结果毁灭性的打击。

p.s.为了防止计算出的概率过小，可以在计算完成后统一乘上相应的系数。分析上面的公式可以看出所有的类条件概率分母都是相同的，所以在这里小白白采用的是只计算分子连乘的形式，平滑处理中，原本的+1改为+0.1。现在上代码~

关于为什么测试集比训练集的类别少一个，显然，那个类别是society，它的分类结果比较差，相当差，特别差，和economy搞混的有很多，于是小白白就舍弃了那个类别的测试集，就是这么任性~舍弃某个类别就要重新测试，因为少了一个类比较概率时就少了一个比较值和可能类，但不需要重新训练了，直接舍弃那个类别训练出来的词典就好了。

终于，分类的结果在我们一步步的摸索中得出了，但是这还没意味着彻底结束了。

首先就像上面说的关于错分类的问题，如果只有少数文章的错分是正常的，毕竟各个类之间的高频词也有交集，但是大范围的错分就需要自己找找原因了，比如说我的关于society类很多被错分到economy类的问题，就是我society类的文章内容与economy有一定的相关度，但是这是一个单向的问题，economy分类的准确率是93%，已经很高了，这说明economy的样本问题不大，主要还是society类样本选取不够典型，类别特征没有完全地显现出来。另外我还遇到一对类别出错的概率也很大，就是military和polity，两个类的准确率都不高，从我们主观理解上来说可能这两个类就比较接近，可想而知它们的词典重合率也比较高，那这就涉及到在收集数据时对都需要哪些类别的规划了，从一开始爬数据就不要选择一些看上去就很相近的两个类别，比如说娱乐和电影、文化历史和读书，这会给后期分类造成很大的困扰。

最后还有一个问题，就是我之前提到的测试执行时间的问题，测试的过程大概花费了4-5个小时，不得不说这个时间太长了。我粗略的估计了一下，数据词典共有8类，对于每一篇预处理过的测试集文章，需要在8个数据词典中都过一遍，设这篇文章有N个词，那么需要对每个词典hash表（约有4-4.5万条词）进行N次遍历找到每个词和它们的词频，记录所有词频相乘的积，而测试集文章大约为8万篇。可以看出这个过程是很费时间的，对词典遍历的效率对总效率的影响最大， 因为这是执行次数最多也是最费时的操作，词典的维数对整个测试程序的执行效率有很重要的影响，so，降维真的真的很重要~虽然小白白还没实践，但是让我们来大胆地预见一下，如果可以把词典降为千级别的维数，测试时间会大大大幅缩短~
